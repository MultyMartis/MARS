# MARS — Dependency Map v0

**Status:** documented — documentation-only cross-system dependency register. This file is not a runtime graph, data pipeline, or machine import; it classifies how MARS **documentation** and **contract** entities relate, per [../AGENTS.md](../AGENTS.md).

**Version:** v0. Supersedes informal “mental model” only when a later version explicitly replaces this map. Revisable per [versioning-model.md](versioning-model.md) and [master-build-map.md](master-build-map.md).

---

## 1. Purpose

- Provide a single, explicit view of how the listed system-level entities depend on one another.
- Reduce silent drift when a contract moves (e.g. signals, registry rows, control-plane surface).
- Block (process-wise) the addition of **Stage 8.5** **Runtime** **Readiness** entities, Self-Heal, Tool Layer, Model Layer, and runtime **documentation** without first recording the relevant rows in this map **and** keeping [risk-register.md](risk-register.md) current for the same change set where **section** **5** / the **Risk** **Register** rules require it (see **section 5** and [master-build-map.md](master-build-map.md) Stage 2 / Stage 8.5 / prerequisite note).

**SAFE UNKNOWN:** This map does not list every file in the repository—only the entity set in **section 4**. Anything not mapped here or not provable is **unknown** for dependency purposes until added.

---

## 2. How to read the map

- Rows are **directed** edges: the `entity_id` (subject) has the stated relationship to `depends_on` (object).
- One entity may have several rows (multiple dependencies).
- `dependency_type` uses the closed vocabulary in **section 3** only; do not invent new types in v0.
- If a link does not fit any type, add a governance note in `logs/lifecycle-log.md` and treat the dependency as **SAFE UNKNOWN** until a future map version extends the type set.

---

## 3. Dependency types (v0 vocabulary)

| Type | Meaning |
|------|--------|
| **requires** | The subject is incomplete, inconsistent, or unusable in documentation terms without the object. |
| **informs** | The subject’s narrative, fields, or consumer expectations are intentionally shaped or constrained by the object. |
| **validates** | The subject (e.g. a self-* operation) checks invariants that the object defines. |
| **blocks** | The subject (often policy / safety doc) forbids or stops certain paths relative to the object; v0 is design-time; future runtime is not in-repo. |
| **logs_to** | A governance or process rule says changes or outcomes should be appended or reflected in the object. |
| **reads_from** | The subject is specified to resolve truth or paths from the object. |
| **writes_to** | The subject is specified to produce or update the object. |
| **governed_by** | The subject’s scope or maintenance rules are set by the object (e.g. authority, ordering). |

## 5. Rules

1. **New system-level entity** — Any new system-level entity in MARS governance or architectural **documentation** must get new rows in **section 4** (and, as needed, an index entry in [master-build-map.md](master-build-map.md) and [README.md](README.md)) in the **same** change set that introduces the entity.
2. **Dependency change** — If a dependency (edge, type, or SoT) changes, update **affected** contracts in this map (reason, `risk_if_broken`, cross-refs), or add a [lifecycle-log.md](../logs/lifecycle-log.md) row with **NEED REVIEW** in the summary (or a dedicated governance convention) to flag incomplete follow-through.
3. **Unknown** — If a dependency is unprovable or unmappable, it is **SAFE UNKNOWN**; state what is unknown; do not guess a `dependency_type`.
4. **Documentation** only — This map does not add build dependencies, configs, or automation; it is **not** a machine dependency graph.
5. **Prerequisite** for **Stage 8.5** **Runtime** **Readiness**, **Self-Heal**, **Tool** / **Model** layers, and **runtime** — Extend or confirm this map **and** review or update [risk-register.md](risk-register.md) per its **§7** rules before **Stage 8.5** **P0** **documentation** **that** **names** **new** **edges**, and before **Tool** (see [master-build-map.md](master-build-map.md) **Stage 8.5**, stages **9–14**) so new entities, edges, and **hazards** are not **implicit** after **Dependency Map** / **Risk Register** v0.
6. **Review** this file when the master build map **materially** changes **stage** order or **gates**.
