# 01-architecture/01-architecture__mars-architecture.md

## mars-architecture.md

# Целевая архитектура MARS

Документ описывает **планируемую целевую** мультиагентную архитектуру **MARS (Multi-Agent Runtime System, «Марс»)**. Реализация по слоям будет наращиваться по фазам дорожной карты; здесь — **единая картина**, без утверждений о том, что все компоненты уже развёрнуты.

## Слои (сверху вниз по ответственности)

### Identity Layer
Идентичность системы, роли пользователей и агентов, контекст «кто что может» на уровне политик (связь с Security/Guardrails).

### Control Plane
Оркестрация: декомпозиция задач, маршрутизация к агентам и моделям, диспетчеризация, состояние workflow, handoff между агентами. Преемник идеи **Main Brain** в стандартной форме.

### Agent Layer
Специализированные агенты (планирование, код, исследование, валидация, память и т.д.) и **Agent Registry**.

### Workflow Layer
Потоки, пайплайны, runbook/playbook, графы задач, state machines, HITL-ветвления.

### Model Layer
Провайдеры LLM, роутинг моделей, адаптеры, fallback, эмбеддинги, rerank, при необходимости vision/speech.

### Tool Layer
Function calling, действия, коннекторы, **Tool Registry**, песочницы, MCP — с жёсткими разрешениями.

### Memory Layer
Краткосрочная и долгосрочная память, эпизодическая/семантическая, рабочая память сессии, память проекта и агента (см. отдельный документ).

### Knowledge / RAG Layer
Базы знаний, индексы, chunking, retrieval, reranking, инъекция контекста, слой цитирования.

### Storage Layer
Операционные БД, векторные хранилища, объектное/файловое хранение, артефакты, аудит, state store — через **адаптеры**.

### Security / Guardrails Layer
Политики, RBAC/ABAC по возможности, одобрение инструментов, валидация ввода/вывода, защита от prompt injection, PII, секреты, sandbox.

### Observability Layer
Трейсинг, логи, метрики, телеметрия, история запусков, лог вызовов инструментов, аудит.

### Evaluation Layer
Evals, регрессии, golden-наборы, сценарные тесты, LLM-as-judge, человеческая ревизия, метрики успеха и галлюцинаций.

### Interface Layer
Чат, админка, CLI, боты, API, webhooks, интеграция с IDE (**Web-GPT, Cursor** как практические каналы).

### DevOps / Runtime Layer
Runtime, воркеры, очереди, планировщик, контейнеры, CI/CD, окружения, секреты, откаты.

## Связь с legacy

Идеи **Main Brain**, **Agent Registry**, **WebGPT Memory Module**, **lifecycle logs**, **project registry**, **workflow trackers**, **SAFE UNKNOWN**, **Git checkpoints**, **Gulp Frontend Agent**, цикл **prompt → execute → report** — отражены в соответствующих слоях как **требования и наследие**, а не как готовая реализация v1.

# 01-architecture/01-architecture__control-plane.md

## control-plane.md

# Control Plane MARS

Целевой **control plane** — центр принятия решений о том, **как** выполняется задача, **каким** агентом и **с какими** инструментами и проверками. Это развитие идеи **Main Brain** в терминах стандартной MAS.

## Компоненты (целевая модель)

| Компонент | Роль |
|-----------|------|
| **Orchestrator / Supervisor** | Общий цикл: принять задачу → согласовать план → запускать шаги → собирать результат; не заменяет специалистов |
| **Planner / Task Decomposer** | Разбиение на подзадачи, зависимости, критерии готовности |
| **Router** | Выбор агента/модели/цепочки по типу задачи и политикам |
| **Dispatcher** | Постановка в очередь, лимиты параллелизма, retry |
| **State Manager** | Снимок состояния workflow, checkpoint’ы логики |
| **Handoff Manager** | Передача контекста между агентами без потери инвариантов и сигналов |

## Возможные технологии (выбор не зафиксирован)

Для графов состояний и мультиагентных потоков в индустрии часто рассматривают:

- **LangGraph**
- **OpenAI Agents SDK**
- **Microsoft Agent Framework / AutoGen**
- **CrewAI**

**Статус для MARS:** технологический стек control plane **НЕ ВЫБРАН** (NOT SELECTED YET). Сравнение и POC — **будущая** работа после исходного пакета и локального starter.

## Границы

Control plane **не** «делает всё сам»: он **декомпозирует**, **маршрутизирует**, **проверяет**, **логирует** и **возвращает** результат через специалистов и инструменты.

## agent-layer.md

# Слой агентов (Agent Layer)

Типы агентов в **целевой** MAS-архитектуре MARS. Конкретный набор и реализация — по мере внедрения; **текущий репозиторий:** каталог агентов — `agents/registry.md`. Таблица-черновик в импортированном пакете по-прежнему в `web-gpt-sources/03_core.md` (заголовок **`# 02-core/02-core__agent-registry.md`** — **legacy source-pack reference, not current repository path**).

## Категории агентов

| Тип | Назначение |
|-----|------------|
| **Specialist Agents** | Узкая экспертиза (домен, стек, формат артефакта) |
| **Worker Agents** | Исполнение рутинных шагов по чёткому контракту |
| **Reviewer Agents** | Ревизия качества, соответствия стандартам |
| **Validator / Guardrail Agents** | Проверки фактов, политик, безопасности (наследие FlyCheck) |
| **Research Agents** | Сбор и синтез информации с оговорками по источникам |
| **Coding Agents** | Код, рефакторинг, интеграции (в т.ч. фронтенд-специалист) |
| **Planning Agents** | Планы, декомпозиция, оценка рисков |
| **Tool Agents** | Обёртки над конкретными инструментами/MCP с минимальной логикой |
| **Memory Agents** | Упаковка/выборка контекста, запись в долговременные хранилища |
| **Critic Agents** | Контрастные проверки, поиск слабых мест плана или ответа |
| **Factory Agent / Agent Builder** | Проектирование и описание новых агентов по стандарту (наследие Factory Engineer) |

## Реестр

**Agent Registry / Agent Catalog** — единая точка правды по **имени, роли, инструментам, ограничениям** и статусу (planned / draft / legacy bridge).

## Связь с другими слоями

Агенты вызываются **Control Plane**’ом, используют **Tool Layer** и **Model Layer**, пишут в **Memory** и **Observability**, подчиняются **Security / Guardrails**.

# 01-architecture/01-architecture__workflow-layer.md

## workflow-layer.md

# Слой workflow (Workflow Layer)

Описывает **оркестрируемые** повторяемые сценарии: от простого «prompt → execute → report» до сложных графов с ветвлениями и HITL.

## Примитивы

- **Workflow** — именованный end-to-end сценарий с входами/выходами и критериями успеха.
- **Flow / Pipeline** — линейная или DAG-последовательность шагов.
- **Runbook / Playbook** — операционные процедуры (часто с человеком в контуре).
- **Task graph** — граф подзадач с зависимостями.
- **State machine** — явные состояния и переходы (удобно для approval, checkpoint).
- **Human-in-the-loop** — паузы на утверждение, выбор ветки, разрешение инструментов.

## Интеграции и аналоги на рынке (справочно)

- **n8n** — внешняя автоматизация и webhook-цепочки.
- **LangGraph** — состояние и ветвления вокруг LLM.
- **CrewAI Flows** — потоки в экосистеме CrewAI.
- **LlamaIndex Workflows** — workflow вокруг индексации и RAG.

Для MARS это **ориентиры**, а не обязательный стек до выбора на фазе проектирования runtime.

## Наследие

**Workflow trackers** и **lifecycle logs** из «Мой первый ИИ system» маппятся на **состояние workflow** и **run history / audit** в Observability + Storage.

# 01-architecture/01-architecture__memory-layer.md

## memory-layer.md

# Слой памяти (Memory Layer)

Целевая модель памяти для MARS — **многоуровневая**, с явными политиками TTL, доступа и происхождения данных.

## Виды памяти (стандартные термины)

| Вид | Описание |
|-----|----------|
| **Short-term / working** | Контекст текущей сессии, «рабочая память» диалога |
| **Conversation memory** | Упорядоченная история реплик и решений в рамках треда |
| **Episodic** | События, запуски, шаги workflow (связь с lifecycle / run history) |
| **Semantic** | Обобщённые знания, факты, извлечённые сущности |
| **Long-term** | Устойчивое хранение между сессиями |
| **Project memory** | Состояние проекта, реестр проектов, артефакты домена |
| **Agent memory** | Предпочтения и паттерны конкретного агента (осторожно: drift, версионирование) |
| **User memory** | Явно согласованные пользовательские факты и предпочтения (PII, compliance) |

## Наследие

Идея **WebGPT Memory Module** — в рамках **project + episodic + semantic** с адаптерами хранения (см. `05-memory/`).

## Будущее

Векторный поиск и RAG — слой **Knowledge / RAG**; память как **запись фактов** и **ссылок на документы** должна оставаться согласованной с политиками **SAFE UNKNOWN**.
