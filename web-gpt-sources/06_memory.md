## memory-architecture.md

# Архитектура памяти MARS

## Источники контекста (наследие + целевое)

| Источник | Роль | Статус |
|----------|------|--------|
| **Markdown-файлы** | Доки, правила, пакеты вроде `web-gpt-sources` | **Уже существующий** актив (файлы в репо/рабочей копии) |
| **Project registry** | Учёт проектов, пути, статусы | Идея из legacy → **план** как системный реестр |
| **Lifecycle logs** | История шагов и запусков | Идея из legacy → **план** как audit / run history |
| **Workflow trackers** | Состояние workflow | Идея из legacy → **план** в State Store |
| **Project state** | Текущие задачи, ветки, договорённости | Частично вручную/Obsidian сегодня → **план** в БД/файлах |
| **Git history** | Фактические изменения кода | **Реальный** источник правды по коду |
| **Vector DB** | Семантический поиск по корпусу | **Будущее** (RAG-слой) |

## Принцип

Память **многоуровневая**: не смешивать «чат» и «источник правды» без версионирования и ссылок.

# Реконструкция контекста (Self-Describe / миграция контекста)

## Идея

**Self-Describe** в legacy-смысле маппится на **introspection / system context**: короткий **сводный пакет**, по которому новый сеанс или агент восстанавливает:

- активный проект и путь;
- релевантные файлы правил;
- последние **сигналы** (`system-signals.md`);
- открытые задачи и статус workflow.

## Миграция контекста

При сигнале **CONTEXT MIGRATION NEEDED**:

1. Зафиксировать **источники** (какие md, какой registry, какие логи).
2. Собрать **минимально достаточный** контекст без утечки лишнего PII.
3. Передать в новый тред/агента с **версией** пакета и датой.

## Статус

Автоматизированный **Memory Agent** для сборки пакета — **планируемый**; до него — ручной чеклист Web-GPT + файлы в репозитории.
