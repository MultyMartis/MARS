# Слой хранения (Storage Layer)

Целевое разделение **типов данных** и **адаптеров** под разные среды (локально, VPS, облако).

## Типы хранилищ

| Тип | Примеры использования |
|-----|------------------------|
| **Operational DB** | Пользователи, проекты, метаданные workflow, ACL |
| **Vector DB** | Эмбеддинги, семантический поиск (будущий RAG) |
| **Object Storage** | Артефакты, бинарники, экспорты |
| **File Storage** | Markdown, конфиги, локальные копии исходников |
| **Artifact Store** | Сборки, отчёты eval, снапшоты контекста (политика retention) |
| **Audit Log Storage** | Неизменяемые или append-only логи аудита |
| **State Store** | Состояние workflow, checkpoint’ы оркестратора |

## Стратегия адаптеров (план)

| Адаптер | Назначение |
|---------|------------|
| **Google Sheets Adapter** | Табличные операции, простые реестры (как в ряде legacy-процессов) |
| **PostgreSQL Adapter** | Основная реляционная БД, при необходимости **pgvector** |
| **Local Files Adapter** | Рабочая копия репозитория, `web-gpt-sources`, Obsidian-совместимые заметки |
| **S3 Adapter** | Объектное хранилище (в т.ч. **S3-совместимое российское**) |
| **Hybrid Storage Adapter** | Комбинация: метаданные в Postgres, файлы локально/S3 |

## Персональные данные (РФ)

При обработке ПДн приоритет: **PostgreSQL на российском VPS**, **локальное файловое хранение**, **S3-совместимое хранилище на российской юрисдикции** — по согласованной политике проекта и договору с оператором. Архитектура MARS **требует** явного выбора региона и модели compliance; детали — вне этого пакета (юридический/ops слой).

# Слой знаний и RAG (Knowledge / RAG Layer)

**Планируемый** слой для работы с документами и извлечением фактов с **цитированием**, а не с «голым» генеративным выдумыванием.

## Компоненты

- **RAG** — retrieval-augmented generation: поиск релевантных фрагментов → сбор контекста → ответ с опорой на источники.
- **Knowledge Base** — корпус документов с метаданными (проект, версия, ACL).
- **Document Index** — структура для быстрого поиска (инвертированный, граф, векторный).
- **Vector Store** — хранение эмбеддингов чанков.
- **Embeddings** — модель и политика обновления индекса.
- **Chunking** — размер, overlap, сохранение структуры (заголовки, код).
- **Retrieval** — фильтры по метаданным, hybrid search.
- **Reranking** — второй этап отбора кандидатов.
- **Context Injection** — безопасная подача в промпт с лимитами токенов.
- **Citation Layer** — явные ссылки на фрагменты источников в ответе.

## Технологии (справочно, не выбрано)

**LlamaIndex**, **LangChain**, **Qdrant**, **Weaviate**, **Chroma**, **PostgreSQL + pgvector** — типичный набор для прототипов и продакшена; выбор стека — **будущая** задача после пилота требований.

## Связь с SAFE UNKNOWN

Если retrieval **не** нашёл подтверждения — ответ должен уходить в режим **SAFE UNKNOWN**, а не заполнять пробелы вымыслом.

## storage-adapters.md

# Адаптеры хранения и память

## Роль слоя адаптеров

Единый интерфейс записи/чтения для **операционных данных**, **файлов**, **таблиц**, **векторов** — с возможностью смены бэкенда без переписывания агентов.

## Приоритет для РФ (требование архитектуры)

- **PostgreSQL** на **российском VPS** для структурированных данных и аудита (после юридического согласования).
- **Локальное файловое хранение** для рабочих копий и части документации.
- **S3-совместимое российское** объектное хранилище — для артефактов при необходимости.

## Существующее vs план

- **Сейчас:** файлы в рабочей копии (в т.ч. этот пакет) — реальный актив.
- **План:** формализованные адаптеры и конфигурация окружений — см. `01-architecture/storage-layer.md`.

# 05-memory/05-memory__rag-strategy.md

## rag-strategy.md

# Стратегия RAG (будущая)

## Цель

Снизить галлюцинации и ускорить ответы за счёт **доказуемых** фрагментов из **Knowledge Base**, с **цитированием** и **rerank**.

## Планируемые шаги (не статус «сделано»)

1. Инвентаризация корпуса (доки, ADR, runbook).
2. Политика chunking и метаданных (проект, версия, ACL).
3. Выбор **vector store** и **embedding** модели (см. architecture knowledge layer).
4. Пайплайн индексации (CI или отдельный job).
5. Слой **citation** в ответах; при отсутствии попаданий — **SAFE UNKNOWN**.

## Связь с WebGPT Memory Module

Идея модуля памяти WebGPT эволюционирует в **project memory + RAG**, без смешения с неструктурированным «простыням» в чате.
